Not All Thoughts Need HBM:
Semantics-Aware Memory Hierarchy for LLM Reasoning

项目计划书

一、研究背景与动机

随着 Chain-of-Thought、o1/o3-style reasoning 等范式的流行，LLM 在推理时会生成大量中间 thought token。这些 token 的 KV-cache 全部驻留在 GPU 的 HBM（High Bandwidth Memory）中，带来三个层面的问题：

  1. 容量瓶颈 — 长链推理的 KV-cache 可达数十 GB，挤占 batch size 空间，降低 serving 吞吐。
  2. 带宽墙 — 每次 decoding step 都要读取全量 KV-cache，memory traffic 与序列长度线性增长，成为延迟瓶颈。
  3. 低利用率 — 实际上大部分中间推理状态在后续生成中几乎不再被 attention 访问，却一直占据最昂贵的存储层。

二、核心观察

并非所有 intermediate thought 对最终答案同等重要。我们通过实证分析发现：

  • Reasoning token 的「后续影响力」（被后续 token attend 的频率、对 output logit 的梯度贡献）呈现长尾分布。
  • 少量关键 thought（如推理链中的关键转折步骤、数值计算结果）贡献了绝大部分信息。
  • 大量「填充性」thought（重复叙述、试探性推导、自我纠正中被放弃的分支）对最终答案贡献极小。

结论：不同 thought 应以不同代价保存，而非统一驻留在最昂贵的 HBM 中。

三、提出方法

3.1 总体思路

借鉴计算机体系结构中经典的 memory hierarchy 思想，我们提出 Semantics-Aware Memory Hierarchy：根据每个中间推理状态对后续生成的语义价值，动态将其分配到不同存储层级——从 HBM 到 host memory、到压缩存储、直至丢弃/重算。目标是在尽量不损失推理质量的前提下，显著降低 HBM 占用、memory traffic 和推理延迟。

3.2 存储层级定义

层级 存储介质 访问代价 放置条件
Tier 0 HBM (on-chip) 最低延迟 高价值 — 被后续 token 频繁 attend 的关键推理步骤、数值结果
Tier 1 DDR / Host Memory PCIe 传输 中等价值 — 偶尔被访问，按需 prefetch 回 HBM
Tier 2 压缩存储 (量化/低秩) 解压开销 低价值但非零贡献 — 压缩后近似恢复，节省带宽
Evict 丢弃 / 按需重算 重算开销 极低价值 — 丢弃后若需则 recompute，大部分情况不再需要

3.3 四个关键组件

  1. Importance Scorer — 在线评估每个 thought segment 的语义重要性。候选信号包括：cumulative attention score、gradient norm、learned lightweight predictor。要求开销极低（< 1% overhead）以避免抵消收益。
  2. Placement Policy — 根据 importance score 与当前内存预算，决定新生成 thought 的初始放置层级。采用阈值分段策略，阈值可由 profiling 统计或在线自适应确定。
  3. Migration Controller — 运行时动态迁移：promotion（低层 → HBM，当被意外频繁访问时）和 demotion（HBM → 低层，当长期未被访问时）。需平衡迁移频率与额外 traffic。
  4. Compression Module — 对 Tier 2 状态执行 KV-cache 量化（如 INT4/INT2）或低秩近似。解压后引入的近似误差需控制在 reasoning quality 可接受的范围内。

3.4 研究目标（量化）

  • 在 HBM 预算降至 30-50% 的条件下，reasoning accuracy 下降 < 2%
  • Memory traffic 减少 40-60%
  • 单条推理延迟降低 20-40%，batch throughput 提升 1.5-2x
  • Importance scorer 的运行时 overhead < 1%

九、第一周：核心假设验证

整个项目建立在一个关键假设上：reasoning token 的 importance 分布是不均匀的，少量关键 token 贡献了大部分信息。第一周的唯一目标就是验证这个假设，同时搭好后续实验的基础设施。

9.1 要验证什么

我们需要回答三个问题：

  1. Importance 分布是否呈长尾？ — 如果 top-20% 的 reasoning token 贡献了 > 60% 的总 importance，说明假设成立，分层存储有意义。
  2. 不同 importance signal 之间是否一致？ — attention-based 和 gradient-based 两种衡量方式如果高度相关（Spearman > 0.6），说明"重要性"是客观可测的，而非度量方式的 artifact。
  3. 低 importance token 真的可以丢弃吗？ — 做一个最简单的 oracle 实验：把 bottom-50% importance 的 KV-cache 直接 mask 掉，看 reasoning accuracy 掉多少。如果掉 < 5%，说明分层存储的收益上界很可观。

9.2 怎么验证

模型与数据：用 DeepSeek-R1-Distill-7B 在 GSM8K 上跑 50-100 条样本。选 7B 是因为可以单卡完成 profiling，GSM8K 是因为答案可自动判对错。

Importance Signal 采集（两条路并行）：

  • Attention-based (A 负责) — 在 transformers 的 attention 层注册 forward hook，逐 decoding step 累加每个 past position 被 attend 的分数。不需要存完整 attention matrix，只存每个 position 的标量累计值。
  • Gradient-based (B 负责) — 固定最终答案 token，反向传播，记录每个 KV position 对 output logit 的梯度 L2 norm。先在几条样本上确认数值稳定且不 OOM。

Oracle Masking 实验 (共同)：

  • 按 importance 排序，mask 掉 bottom-K%（K = 30, 50, 70）的 KV position（attention score 置零）
  • 让模型在 masked KV-cache 上继续生成答案
  • 对比 accuracy：如果 mask 50% 后 accuracy 只掉 1-3%，这就是论文 Sec 3 Motivation 最有力的数据点

9.3 基础设施搭建

和验证实验并行，把后续几周需要的骨架搭好：

  • A：建 GitHub 仓库 + conda 环境 + 目录结构；下载模型权重；准备 benchmark 数据集（GSM8K, MATH, GPQA, ARC-Challenge, HumanEval 统一 JSONL 格式）
  • B：文献调研（CoT/DeepSeek-R1/QwQ 的 reasoning 机制 + FlashAttention/FlexGen 的 memory 分析方法），整理对比笔记

9.4 周末检查点

第一周结束时两人对齐，需确认：

  • Importance 分布图是否支持长尾假设（定量：top-20% 覆盖多少 %）
  • Attention signal 和 gradient signal 的相关性
  • Oracle masking 实验的 accuracy 下降曲线
  • 基于以上结果，决定：主力 importance signal 用哪种；是否需要调整技术路线

如果假设验证失败（分布均匀 / masking 后 accuracy 大幅下降），需要在第二周前讨论是否转向 temporal locality 等替代方向。
